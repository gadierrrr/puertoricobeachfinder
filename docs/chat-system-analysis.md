## Database Schema

```sql
-- Chat rooms (global, per-beach, or regional)
CREATE TABLE chat_rooms (
    id TEXT PRIMARY KEY,
    room_type TEXT NOT NULL CHECK(room_type IN ('global', 'beach', 'region')),
    beach_id TEXT REFERENCES beaches(id) ON DELETE CASCADE,
    region_code TEXT,  -- 'north', 'south', 'east', 'west', 'islands'
    name TEXT NOT NULL,
    description TEXT,
    is_active INTEGER DEFAULT 1,
    message_count INTEGER DEFAULT 0,
    last_message_at TEXT,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(room_type, beach_id),
    UNIQUE(room_type, region_code)
);

-- Chat messages
CREATE TABLE chat_messages (
    id TEXT PRIMARY KEY,
    room_id TEXT NOT NULL REFERENCES chat_rooms(id) ON DELETE CASCADE,
    user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    content TEXT NOT NULL,
    beach_mention_id TEXT REFERENCES beaches(id),  -- Optional @beach link
    reply_to_id TEXT REFERENCES chat_messages(id), -- Thread support
    is_pinned INTEGER DEFAULT 0,
    is_hidden INTEGER DEFAULT 0,
    hidden_reason TEXT,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT
);

-- Indexes for performance
CREATE INDEX idx_messages_room_time ON chat_messages(room_id, created_at DESC);
CREATE INDEX idx_messages_user ON chat_messages(user_id);
CREATE INDEX idx_messages_beach_mention ON chat_messages(beach_mention_id)
    WHERE beach_mention_id IS NOT NULL;
CREATE INDEX idx_rooms_beach ON chat_rooms(beach_id) WHERE beach_id IS NOT NULL;
CREATE INDEX idx_rooms_type ON chat_rooms(room_type);

-- User chat preferences & muting
CREATE TABLE chat_user_settings (
    user_id TEXT PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    muted_rooms TEXT,  -- JSON array of room_ids
    notifications_enabled INTEGER DEFAULT 1,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP
);

-- Message reactions (optional, lightweight engagement)
CREATE TABLE chat_reactions (
    id TEXT PRIMARY KEY,
    message_id TEXT NOT NULL REFERENCES chat_messages(id) ON DELETE CASCADE,
    user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    reaction TEXT NOT NULL CHECK(reaction IN ('ğŸ‘', 'ğŸŒŠ', 'â˜€ï¸', 'ğŸ™', 'ğŸ˜')),
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(message_id, user_id, reaction)
);

-- Moderation log
CREATE TABLE chat_mod_actions (
    id TEXT PRIMARY KEY,
    message_id TEXT REFERENCES chat_messages(id),
    user_id TEXT REFERENCES users(id),  -- User being moderated
    mod_user_id TEXT NOT NULL REFERENCES users(id),  -- Admin
    action TEXT NOT NULL CHECK(action IN ('hide', 'unhide', 'pin', 'unpin', 'warn', 'mute')),
    reason TEXT,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP
);

-- Initialize global room
INSERT INTO chat_rooms (id, room_type, name, description)
VALUES (
    'global',
    'global',
    'Beach Community',
    'General chat for all Puerto Rico beach lovers'
);
```

## Summary & Recommendation

**Recommended Approach: Hybrid Model with HTMX Polling**

1. **Beach-specific chat rooms** integrated into each beach page
2. **Single global "Community" room** for general discussion
3. **HTMX polling** every 3-5 seconds (fits existing architecture)
4. **Beach @mentions** in global chat link to beach pages
5. **Phased rollout** starting with beach chats only

**Why this works for Beach Finder:**
- Beach conditions don't change by the second (3s polling is fine)
- Users are already on beach pages (natural chat context)
- Builds on existing HTMX patterns (no new infrastructure)
- Moderation scope is manageable (focused on beaches)
- Drives registration (chat requires auth to post)

**Start with Phase 1 (MVP)** and iterate based on usage patterns.
